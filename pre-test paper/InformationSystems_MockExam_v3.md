# 原卷风格总结

- 历年试卷结构高度稳定：采用“单选(10)+多选(5)+简答(2)+实务(3)”的标准模型，侧重基础面覆盖与系统工程实践能力的平衡。
- 命题特征明确：高度场景化，紧贴政企、事业单位及公共服务（如物流、校园、医疗、城市管理）的信息化真实业务需求。
- 技术主轴统一：核心考点长期锁定在计算机网络、数据库设计与SQL优化、软件工程生命周期、信息安全防护以及算法逻辑。
- 演进趋势：近年显著增加了对新技术（AIGC、云计算、物联网等）在应用层面的理解考查，且实务题的综合度、跨模块分析能力要求持续提升。

# 本次新试卷设计思路

- 结构严控：严格遵循 10-5-2-3 题型分布，分值权重与难度分布向“实务分析”倾斜。
- 场景创新：完全避开已有的垃圾分类、养老、文旅、实验室、冷链、应急指挥等场景，采用“社区志愿者积分系统”、“工业园区能耗监测”、“智慧医院网络安全”三个全新领域。
- 命题升级：
    - 客观题：引入更多“系统现象诊断”与“指标推理”题干，要求考生根据 CPU、I/O、连接数、丢包率等数据点进行多步推理。
    - 主观题：强化“系统一致性控制（事务原子性）”、“复杂数据聚合（SQL 嵌套与关联）”、“告警抑制算法”及“多网分区安全策略”的考查。
- 区分度设计：确保每个客观题都有“看似合理”的次优方案作为干扰项，区分“死记硬背型”与“原理理解型”考生。

# 难度控制说明

- 基础题（30%）：涵盖标准网络掩码、基础 SQL 语法、软件生命周期模型。
- 中等难度题（50%）：要求结合特定场景进行性能排查、子网规划推理、等价类综合设计、多表关联统计。
- 高区分度题（20%）：集中在实务题的并发一致性方案设计、复合索引最优选择、复杂拓扑的互访策略制定。
- 总体预期：通过提供具体的约束条件（如“不扩容硬件前提下优化”、“连续3次触发判定”），迫使考生进入“工程决策”状态。

## 四、实务题

### 1. 社区志愿者服务与积分兑换系统设计题

某街道办拟建设“社区志愿者服务与积分兑换系统”。志愿者通过小程序报名参加社区服务活动（如慰问老人、垃圾分类宣传等）。活动结束后，社区管理员记录志愿者服务时长，系统根据 1 小时 = 10 积分的规则自动为志愿者增加积分。志愿者可以使用积分在积分商城中兑换生活用品（如抽纸、食用油等）。

已知主要数据项如下：
- 志愿者：身份证号、姓名、联系电话、累计积分、注册日期。
- 社区活动：活动编号、活动名称、所属社区、计划时间、人数上限。
- 服务记录：记录编号、身份证号、活动编号、签到时间、服务时长（小时）。
- 积分商品：商品编号、商品名称、所需积分、库存数量。
- 兑换订单：订单编号、身份证号、商品编号、兑换时间、扣减积分、订单状态。

业务规则如下：
1. 一个志愿者可参加多次活动，但同一活动限报名一次。
2. 志愿者积分不足以兑换目标商品时，兑换操作应被拒绝。
3. 兑换成功必须同时完成：扣减个人积分、创建兑换订单、减少商品库存。

根据以上材料，回答下列问题：
1. 设计该系统的主要实体及联系，写出对应的关系模式，并标明各模式的主键。
2. 编写 SQL 语句，查询 `2026-03` 月份期间，累计服务时长排名前 10 的志愿者信息。要求输出姓名、联系电话、总服务时长，并按总时长降序排列。
3. 设计“积分兑换商品”的核心处理流程。请说明在并发环境下，如何确保“积分不超支”和“库存不超卖”，可用文字逻辑或伪代码表达，并必须体现数据库事务的应用。

### 2. 工业园区智慧能耗监测系统设计题

某工业园区部署了一套能耗监测系统，通过智能电表实时采集各厂房的用电数据。系统需要对异常用电行为（如漏电风险、违规大功率电器使用）进行自动监测与告警。

已知数据结构如下：
- 电表：表编号、所属厂房、安装位置、额定电压上限、额定电流上限。
- 实时采集：记录 ID、表编号、采集时间、当前电压、当前电流、瞬时功率、当日累计电量。
- 告警规则：规则编号、监测指标（如电流）、判定条件（如 > 110% 额定值）、告警等级。
- 告警日志：日志 ID、表编号、采集记录 ID、告警时间、告警内容、处理状态。

业务逻辑：
1. 系统每 5 分钟采集一次数据。
2. 为了防止因为传感器瞬时扰动导致的虚假告警，系统要求：仅当某个指标“连续 3 次”超过阈值时，才触发正式告警并记录日志。

根据以上材料，回答下列问题：
1. 写出该系统的关系模式设计（至少包含电表、实时采集、告警日志），并标明主键。
2. 编写 SQL 语句，查询某厂房（假设 ID 为 `A01`）内，最近 24 小时内所有瞬时功率超过 500kW 的采集记录。要求输出表编号、采集时间、当前电流、瞬时功率。
3. 针对“连续 3 次超限才告警”的需求，设计一个逻辑处理方案。要求说明如何保存中间状态，如何判定连续性，以及在触发告警后如何避免在下一次采集时（如果仍然超限）产生重复冗余告警。

### 3. 三甲医院互联网门诊网络与数据安全设计题

某三甲医院拟建设“互联网门诊”系统，支持患者在线咨询、视频问诊、处方流转及医保在线结算。该院现有机房分为：内网（核心 HIS/LIS/PACS 系统，存储患者病历与敏感隐私）、互联网区（承载医院官网及挂号预约）。本次建设要求：
1. 互联网问诊系统需与互联网连通，但不能直接接触内网核心数据库。
2. 问诊过程中产生的电子处方需从互联网区安全同步到内网 HIS 系统进行药房配药。
3. 内部医生在内网终端上进行问诊操作。
4. 符合国家等保三级（等保 3.0）安全要求，具备防攻击、防泄密和全量审计能力。

根据以上材料，回答下列问题：
1. 请画出或详细描述该系统的网络安全逻辑架构，至少包含：互联网接入区、DMZ 区（互联网医院应用）、核心内网区、运维管理区。并说明各区域之间应通过哪些安全设备进行物理或逻辑隔离。
2. 在内外网数据交换（如电子处方同步）环节，建议采用什么技术手段或设备（如物理隔离网闸、前置机、堡垒机等）来保证安全？请简述其工作原理。
3. 视频问诊业务对实时性要求极高。若患者反馈“视频画面频繁卡顿、延迟严重”，而监控显示服务器资源（CPU/内存）充足。请从网络带宽、QoS（服务质量）策略、CDN 应用及网络路径优化三个维度提出具体的排查与优化方案。
